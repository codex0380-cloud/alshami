# مراجعة مشروع شاورما الشامي

هذا الملف يلخص أهم النواقص والمخاطر في المشروع الحالي، سواء من ناحية الأمان أو التقنية أو المميزات التشغيلية.

## ملخص سريع

المشروع عبارة عن موقع طلبات ثابت `static` لمطعم، يعتمد على Firebase Realtime Database، وفيه:

- صفحة رئيسية للعميل.
- سلة وطلب أونلاين.
- صفحة تتبع الطلب.
- لوحة إدارة للطلبات.
- Service Worker وملف Manifest لتجربة PWA بسيطة.

المشروع شكله مكتمل من ناحية الواجهة، لكن يحتاج تحسينات مهمة قبل الاعتماد عليه كنظام طلبات حقيقي.

## أولويات عاجلة

1. إغلاق قواعد Firebase ومنع الكتابة العامة.
2. منع حقن HTML/JavaScript في بيانات الطلبات.
3. نقل حساب الأسعار والإجمالي والخصومات إلى سيرفر أو Cloud Functions.
4. عدم إظهار نجاح الطلب للعميل إذا فشل الإرسال إلى Firebase.
5. تحسين توليد رقم الطلب لمنع التكرار أو استبدال طلب قديم.

## مشاكل أمنية

### 1. قواعد Firebase مفتوحة للكتابة

في `firebase-rules.json` توجد قواعد مثل:

```json
".write": true
```

هذا يعني أن أي شخص يستطيع الوصول إلى قاعدة البيانات يمكنه:

- إضافة طلبات وهمية.
- تعديل طلبات موجودة.
- حذف أو تغيير حالات الطلبات.
- إرسال spam إلى `subscribers` و`errors`.

المطلوب:

- السماح للعميل بإنشاء طلب جديد فقط.
- منع العميل من تعديل أو حذف طلب بعد إنشائه.
- السماح للأدمن فقط بقراءة وتعديل وحذف الطلبات.
- إضافة قواعد تحقق `validate` للحقول المطلوبة.

### 2. لوحة الإدارة ليست حماية كافية وحدها

تسجيل الدخول موجود في `admin.js` باستخدام Firebase Auth، لكن الحماية الحقيقية يجب أن تكون في قواعد Firebase نفسها.

كذلك يوجد fallback password داخل `admin.js`:

```js
let ADMIN_PASSWORD = 'shami2024';
```

المطلوب:

- حذف أي كلمة سر مكتوبة داخل ملفات الواجهة.
- الاعتماد فقط على Firebase Auth.
- تقييد صلاحيات قاعدة البيانات حسب المستخدم المسجل.

### 3. احتمال Stored XSS

المشروع يستخدم `innerHTML` لعرض بيانات قادمة من الطلبات مثل:

- اسم العميل.
- العنوان.
- الملاحظات.
- أسماء الأصناف.

لو شخص أرسل طلب يحتوي على HTML أو JavaScript، قد يظهر الكود داخل لوحة الإدارة أو صفحة التتبع.

المطلوب:

- استخدام `textContent` عند عرض بيانات المستخدم.
- أو عمل دالة escape لكل نص قبل إدخاله في HTML.
- عدم إدخال بيانات المستخدم مباشرة داخل `innerHTML`.

### 4. السعر والإجمالي محسوبان من جهة العميل

العميل يرسل الطلب مباشرة إلى Firebase، ومعه:

- `subtotal`
- `deliveryFee`
- `discount`
- `total`
- `coupon`

أي مستخدم يمكنه تعديل هذه القيم من المتصفح قبل الإرسال.

المطلوب:

- حفظ المنيو والأسعار في مصدر موثوق.
- إرسال ids والكميات فقط من العميل.
- حساب الإجمالي النهائي داخل Cloud Function أو Backend.

### 5. تسجيل الأخطاء مفتوح

المسار `errors` يسمح بالكتابة العامة. هذا مفيد للتتبع، لكنه ممكن يتحول إلى spam أو تخزين بيانات غير مرغوبة.

المطلوب:

- إضافة rate limiting أو App Check.
- تحديد شكل البيانات المسموح بها.
- عدم تخزين بيانات حساسة مثل userAgent الكامل أو روابط تحتوي بيانات شخصية إلا عند الحاجة.

## مشاكل تشغيلية

### 1. نجاح وهمي عند فشل الطلب

في حالة فشل إرسال الطلب إلى Firebase، الكود يعرض رسالة نجاح للعميل. هذا قد يؤدي إلى ضياع طلبات حقيقية.

المطلوب:

- عند فشل الإرسال، عرض رسالة خطأ واضحة.
- توفير زر إعادة المحاولة.
- عدم مسح السلة إلا بعد نجاح الحفظ.

### 2. رقم الطلب قابل للتكرار

رقم الطلب 4 أرقام فقط باستخدام `Math.random`. هذا قد يسبب تكرار، خصوصا مع نفس رقم الهاتف، لأن مفتاح الطلب مبني من:

```js
phone + '_' + orderNumber
```

المطلوب:

- استخدام Firebase push id كمفتاح أساسي.
- توليد رقم طلب قابل للعرض فقط، مع ضمان عدم التكرار.
- يمكن استخدام تاريخ اليوم + عداد يومي أو رقم أطول.

### 3. الدفع بالكارت غير مفعل

تم حذف اختيار الدفع بالكارت من الواجهة الحالية لأن المشروع لا يحتوي على بوابة دفع فعلية.

المطلوب مستقبلا:

- عدم إعادة خيار الكارت إلا بعد إضافة payment gateway حقيقية.
- إبقاء طرق الدفع الحالية واضحة: كاش أو Vodafone Cash.
- إضافة حالة دفع منفصلة عن حالة الطلب عند تفعيل الدفع الإلكتروني.

### 4. تتبع الطلب يعتمد على رقم الهاتف ورقم الطلب

صفحة التتبع تستخدم `phone + orderNumber` للوصول المباشر للطلب. هذا أفضل من عرض كل الطلبات، لكنه ما زال يحتاج قواعد Firebase تمنع قراءة بيانات غير مصرح بها.

المطلوب:

- عدم جعل كل طلب قابل للقراءة مباشرة بدون تحقق.
- التفكير في token خاص للتتبع يتم توليده وقت إنشاء الطلب.

## تقنيات ناقصة

### 1. لا يوجد Build Pipeline

المشروع لا يحتوي على `package.json` أو tooling واضح.

المطلوب:

- إضافة `package.json`.
- إضافة أدوات lint/format.
- تجهيز build production.
- تقليل الاعتماد على CDN runtime في الإنتاج.

### 2. Tailwind يعمل من CDN داخل المتصفح

استخدام Tailwind browser build مناسب للتجربة، لكنه ليس الأفضل للإنتاج.

المطلوب:

- بناء CSS مسبقا.
- تصغير الملفات.
- إزالة الاعتماد على compiler داخل المتصفح.

### 3. لا توجد اختبارات

لا توجد اختبارات للطلب أو الحسابات أو صلاحيات Firebase.

المطلوب:

- اختبارات لحساب السلة والإجمالي والخصومات.
- اختبارات لقواعد Firebase إن أمكن.
- اختبار يدوي موثق لمسار الطلب من البداية للنهاية.

### 4. لا توجد Security Headers

المشروع يحتاج إعدادات استضافة تضيف:

- `Content-Security-Policy`
- `X-Content-Type-Options`
- `Referrer-Policy`
- `Permissions-Policy`

المطلوب:

- إضافة هذه الهيدرز في إعدادات الاستضافة مثل Firebase Hosting أو Netlify أو Vercel.

### 5. لا يوجد Firebase App Check

Firebase App Check يقلل إساءة استخدام قاعدة البيانات من خارج الموقع.

المطلوب:

- تفعيل App Check.
- ربطه بالموقع.
- مراقبة الطلبات غير الشرعية قبل فرضه بالكامل.

## مميزات ناقصة مقترحة

### لوحة الإدارة

- إدارة المنيو والأسعار من لوحة الإدارة.
- إيقاف صنف مؤقتا عند نفاده.
- إدارة العروض والكوبونات.
- صلاحيات متعددة: مالك، كاشير، مطبخ، دليفري.
- سجل تعديلات لكل طلب.
- طباعة إيصال محسنة.
- صفحة مطبخ منفصلة للطلبات الجديدة فقط.

### الطلبات

- أقل حد للطلب.
- رسوم توصيل حسب المنطقة.
- اختيار منطقة التوصيل من قائمة.
- منع الطلب خارج مواعيد العمل.
- وقت تحضير تقديري.
- إعادة الطلب السابق.
- ملاحظات لكل صنف وليس للطلب كله فقط.

### الدفع

- تم إلغاء خيار الدفع بالكارت من الواجهة الحالية إلى أن توجد بوابة دفع حقيقية.
- Vodafone Cash مع رقم تحويل أو صورة إثبات.
- بوابة دفع حقيقية إذا لزم لاحقا.
- حالة دفع منفصلة عن حالة الطلب.

### الإشعارات

- إشعار صوتي ومرئي للأدمن موجود، لكن يمكن تحسينه.
- إشعارات Push عبر Firebase Cloud Messaging.
- إرسال إشعار Telegram أو WhatsApp Business عند الطلب الجديد.
- إشعار للعميل عند تغيير الحالة.

### التقارير

- تقرير يومي.
- تقرير أسبوعي وشهري.
- أكثر الأصناف مبيعا.
- متوسط قيمة الطلب.
- الطلبات الملغية وأسباب الإلغاء.
- تصدير CSV حسب فترة زمنية.

### تجربة العميل

- تحسين رسائل الخطأ.
- توضيح مناطق التوصيل.
- عرض حالة المطعم: مفتوح أو مغلق.
- حفظ بيانات العميل محليا لتسهيل الطلب التالي.
- تحسين SEO باستخدام روابط وصور مطلقة في Open Graph.

## خطة تنفيذ مقترحة

### المرحلة 1: أمان عاجل

- تعديل قواعد Firebase.
- إزالة كلمة السر من الكود.
- منع XSS عند عرض بيانات المستخدم.
- إصلاح رسالة النجاح عند فشل إرسال الطلب.

### المرحلة 2: سلامة الطلبات

- تحسين توليد أرقام الطلب.
- نقل حساب الإجمالي للسيرفر.
- تقييد الكوبونات والتحقق منها من مصدر موثوق.
- إضافة تحقق قوي للبيانات.

### المرحلة 3: تشغيل وإدارة

- إضافة إدارة المنيو من لوحة الأدمن.
- إضافة مناطق ورسوم توصيل.
- تحسين التقارير والتصدير.
- إضافة إشعارات حقيقية للأدمن.

### المرحلة 4: جودة وإنتاج

- إضافة `package.json`.
- تجهيز build production.
- إضافة linting وformatting.
- إضافة Security Headers.
- تفعيل App Check.
- إضافة اختبارات أساسية.

## ملاحظات أخيرة

المشروع جيد كواجهة وتسلسل استخدام، لكنه حاليا لا يصلح كنظام طلبات production بدون إغلاق Firebase ومعالجة XSS وسلامة الأسعار. أفضل بداية هي تأمين قاعدة البيانات وطريقة إنشاء الطلبات، ثم تطوير المميزات الإدارية بعد ذلك.
